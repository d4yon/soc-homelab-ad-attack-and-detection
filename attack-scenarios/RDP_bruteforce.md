# Scenario 1 — RDP Bruteforce Attack

**MITRE ATT&CK:** [T1110.001](https://attack.mitre.org/techniques/T1110/001/) — Brute Force: Password Guessing  
**Tactic:** Initial Access / Credential Access  
**Severity:** High  
**Tools:** NetExec, Hydra  
**Target:** WRKST-01-WIN10 (domain-joined workstation, netwatch.local)

---

## 1. Attack Description

Adversaries may use brute force techniques to gain access to accounts when passwords are unknown. RDP (Remote Desktop Protocol, TCP 3389) is one of the most targeted services on the Internet — organizations that expose RDP directly without a VPN create a trivially exploitable attack surface.

This scenario simulates a realistic Internet-based RDP bruteforce attack: an external attacker discovers an exposed RDP endpoint, enumerates information from the login page, and systematically attempts credentials until a valid combination is found.

**Real-world context:** Shodan indexes over 2 million exposed RDP endpoints globally. An attacker can identify targets in minutes using passive recon alone — no active scanning required. The presence of a port on Shodan does not mean it is exploitable (NLA, MFA, lockout policies, and strong passwords all reduce risk significantly), but misconfigured endpoints are trivially compromised.

## 2. Why It Matters

- RDP bruteforce is consistently listed in top exploited techniques year over year — [Kaspersky](https://www.kaspersky.com/about/press-releases/the-great-migration-of-cyberthreats-attacks-on-remote-desktop-protocols-grew-by-242-reaching-33-billion-in-2020) directly links the 242% surge in RDP brute force attacks in 2020 (3.3 billion attempts in eleven months) to the mass shift to remote work driven by COVID-19, as organizations rushed to expose RDP to enable work-from-home, often with poor configuration and no MFA
- A successful bruteforce grants interactive access to a domain-joined machine — an attacker can immediately pivot to credential dumping and lateral movement
- Detection requires correlation logic, not just individual event monitoring — a single failed logon is noise, a pattern of failures from the same source is an attack
- The difference between a user mistyping their password and an automated bruteforce campaign is visible in the logs — this scenario demonstrates exactly that

**As a SOC analyst:** you need to understand not just that an alert fired, but *why* — what the underlying Windows events mean, how lockout policies affect log generation, and what changes in the logs once a bruteforce transitions from "attempting" to "account locked out."

## 4. Reconnaissance
### 4.1 Passive recon — Shodan

Before touching the target, a real attacker uses passive reconnaissance to identify exposed RDP endpoints.

<img width="1527" height="631" alt="image" src="https://github.com/user-attachments/assets/c95150c1-4144-4547-8d8e-4aaff14a38b9" />

Shodan exposes the following information without any active scanning:
- Public IP of the target
- OS version (extracted from the NTLM banner)
- Hostname (extracted from the RDP SSL certificate Common Name)
- SSL certificate details (self-signed = likely unmanaged endpoint)

The SSL certificate visible on Shodan is the self-signed TLS certificate auto-generated by Windows for the RDP service. It encrypts the transport layer but does **not** protect against bruteforce — credentials are still challenged over the encrypted channel. The certificate Common Name leaks the machine hostname, which is useful targeting information for an attacker.

### 4.2 Active recon — Nmap

```bash
nmap -p 3389 -sV --script rdp-enum-encryption 10.2.5.100
```

<img width="762" height="302" alt="image" src="https://github.com/user-attachments/assets/92e3d43b-9d31-4e7e-b238-a28979277361" />

The `--script rdp-enum-encryption` script connects to the RDP service and tests which security layers are supported:

```
| rdp-enum-encryption:
|   Security layer
|     CredSSP (NLA): SUCCESS
|     CredSSP with Early User Auth: SUCCESS
|_    RDSTLS: SUCCESS
```

From a reconnaissance perspective, what is of particular interests to us is NLA. The nmap output tells us which security protocols the target **supports** — not which ones are actively enforced. `CredSSP (NLA): SUCCESS` means the server is capable of NLA, nothing more.

**What is NLA?**  
Network Level Authentication is a security layer that requires the client to authenticate _before_ a full RDP session is established. Without NLA, the RDP session opens first and credentials are challenged inside it — meaning the login page (and everything it exposes) is reachable without any authentication. With NLA enforced, the server challenges credentials at the network level before the session even opens, blocking unauthenticated access to the login page entirely.

**Why interesting from a recon perspective?**  
If NLA is enforced, an attacker cannot reach the login page — no hostname confirmation, no last username, no screenshot. If it's disabled, all of that is available pre-authentication. Knowing whether NLA is actually enforced directly determines what information the attacker can gather and how they approach the bruteforce.

The problem is that nmap can't answer that question — `CredSSP (NLA): SUCCESS` is the same whether NLA is required or just available. To know if NLA is actually enforced, we need a tool that attempts a connection and checks whether credentials are demanded before the session opens. That's where NetExec comes in.

> 💡 **Insight:** The login page leaks the last logged-on username by default — a Windows behavior that can be suppressed via GPO (`Interactive logon: Do not display last user name`). In this scenario, it directly provides the attacker with a valid username to target, eliminating the need for user enumeration.

> 💡 **Insight — NLA before/after:** With NLA enabled, an attacker cannot reach the login page. With NLA disabled, the full session is exposed pre-authentication. This is why NLA should always be enforced on Internet-exposed RDP endpoints.

> 💡 **Insight — Nmap vs NetExec on NLA:** Nmap reports `CredSSP (NLA): SUCCESS` even after NLA is disabled because it tests whether the *protocol is supported*, not whether it is *enforced*. NetExec correctly reports `nla:False` because it tests whether the server *requires* NLA authentication. These are different questions — Nmap tells you what the server can do, NetExec tells you what the server demands.

### 6.2 Event Analysis

#### Event 4625 — Failed Logon

<img width="962" height="662" alt="image" src="https://github.com/user-attachments/assets/e4286b39-d0f7-4c68-8ede-4057c2368404" />

Each failed RDP attempt generates a 4625 with:
- **LogonType: 3** (Network) — this is the logon type generated by NetExec bruteforce over RDP
- **SubStatus: 0xC000006A** — wrong password 

From these events we can also extract the **targeted account** and the **source IP** of the authentication attempts.

In our attack demo, a single Event ID 4625 can trigger two distinct Wazuh alerts depending on context:

**Rule 60122 — Logon Failure, Unknown user or bad password**

- Level : 5
- Fires on every individual 4625 event
- MITRE : T1531 — Account Access Removal / Impact

<img width="750" height="595" alt="image" src="https://github.com/user-attachments/assets/79574e75-ace2-41c0-816d-17fd649e1942" />

**Rule 60204 — Multiple Windows Logon Failures**

- Level : 10
- Fires after **8 consecutive** 4625 events (built-in frequency threshold)
- MITRE : T1110 — Brute Force / Credential Access

<img width="786" height="554" alt="image" src="https://github.com/user-attachments/assets/3a166ffe-1966-4f34-bf53-9033b3e2492d" />

Rule 60204 is particularly interesting from an analyst perspective — it is Wazuh's built-in bruteforce detection rule, and it triggers after **8 consecutive failures**. This raises a valid question: why 8, and is that threshold appropriate?

In practice, 8 failed attempts is a relatively high bar. A determined attacker using a short wordlist could stay deliberately under that threshold, or adopt a password spraying approach — cycling through multiple accounts with a single password at a time to avoid account lockouts and frequency-based detection. This highlights the importance of custom rule tuning. Built-in rules provide baseline coverage, but adapting thresholds and correlation logic to your specific environment is what makes detection meaningful. Password spraying detection and low-and-slow bruteforce evasion techniques are areas worth exploring further as a natural extension of this scenario.

> 💡 **Insight — LogonType 3 vs 10:** Standard RDP sessions generate LogonType **10** (RemoteInteractive). NetExec bruteforce generates LogonType **3** (Network) because it uses a lower-level authentication negotiation rather than a full interactive session. This distinction is useful for detection tuning — a rule targeting LogonType 10 would miss this attack vector.

> 💡 **Insight — SubStatus:** Before lockout, 4625 carries SubStatus `0xC000006A` (wrong password). This is the value rules 60122 and 60204 are effectively acting on. The SubStatus changes after lockout — addressed in the remediation section.

#### Event 4624 — Successful Logon 

<img width="932" height="408" alt="image" src="https://github.com/user-attachments/assets/7d1edc0f-99fb-4431-980b-5937eca9cc68" />

At the Windows event level, filtering exclusively on the workstation agent means we are missing critical information — notably the source IP and full account details, as authentication for domain accounts is ultimately validated by the DC. Checking the DC agent alongside the workstation agent would give a more complete picture, and is something worth keeping in mind for the subsequent scenarios.

Wazuh's built-in rule 92652 flags this successful logon with multiple MITRE techniques simultaneously — Pass the Hash (T1550.002) and Domain Accounts (T1078.002) — and maps it across a broad set of tactics: Defense Evasion, Lateral Movement, Persistence, Privilege Escalation, and Initial Access. This is an interesting contrast with the raw Windows event, which simply records a successful login with no further context. Wazuh enriches the same event with threat intelligence context, treating an NTLM network logon as a potential attack vector rather than routine activity. Whether it actually is one cannot be determined from this event alone — which is precisely the point: correlation across multiple events and agents is what separates a meaningful alert from noise.

<img width="1115" height="560" alt="image" src="https://github.com/user-attachments/assets/89615afc-121d-4e50-ba92-3867ada52a01" />

> 💡 **Insight:** The rule 92652 description explicitly mentions *"possible pass-the-hash attack"* for a standard NTLM network logon — this illustrates that NTLM Type 3 logons are inherently ambiguous. Wazuh cannot distinguish between a legitimate NTLM logon and a PtH attack from the 4624 event alone. This is exactly why Scenario 3 (Pass-the-Hash) requires correlation with Scenario 2 (Mimikatz) to be meaningful.

### 6.4 Why NetExec over Hydra

Both tools can bruteforce RDP, but NetExec is purpose-built for Active Directory environments while Hydra is a generic bruteforce tool. In this context NetExec is the more relevant choice for three reasons:

- **AD-aware** — understands NTLM, Kerberos, SMB, WinRM natively and can enumerate users, shares, and policies in addition to credential testing
- **Broader capability** — supports Pass-the-Hash, Pass-the-Ticket, lateral movement, and command execution across multiple hosts — it's a complete AD offensive tool, not just a password sprayer
- **Stealthier** — during testing, Hydra generated 4776 events at the local level and exposed the attacker's machine name in the logs while NetExec did not

<img width="1788" height="646" alt="image" src="https://github.com/user-attachments/assets/65c1e71e-c591-426b-804e-03f2f5bf1ef8" />

> 💡 **Insight — Hydra vs NetExec stealth:** When using NetExec, Event 4776 is generated at the DC level during NTLM credential validation — it does not appear when filtering on the workstation agent. When using Hydra, the picture is different: because Hydra also tests local authentication, 4776 events are generated directly on the workstation and appear alongside the 4625 events in the workstation agent logs. Additionally, Hydra exposes the attacker's machine hostname in the 4625 events, while NetExec does not. Both factors combined — additional event noise and hostname exposure — make Hydra a significantly louder tool from a detection standpoint, and NetExec the stealthier choice.

> 💡 **Insight** From a defensive perspective, the absence of a workstation name in authentication logs is itself a suspicious indicator worth alerting on. NTLM authentication events with no associated workstation name should be treated as a suspicious indicator and flagged for review.

## 7. Incident Response

When a bruteforce alert fires, the immediate priority is to stop the bleeding — cut off the attacker's access, assess whether the attack succeeded, and preserve evidence for further investigation. The following actions simulate a first-responder SOC workflow.

---

### 7.1 Block the attacker IP in pfSense

The first action is to prevent any further connection attempts from the identified source IP. In a real environment this would typically be handled by an automated response or a firewall management platform, but the principle is the same — isolate the source before investigating further.

```
Firewall → Rules → WAN → Add rule
  Action      : Block
  Protocol    : any
  Source      : 10.2.5.150 (KALI-ATK)
  Destination : any
  Description : Block attacker RDP bruteforce
```

<img width="1168" height="455" alt="image" src="https://github.com/user-attachments/assets/bfbeb6c4-fd43-429b-b7f6-78b562a0add9" />

---

### 7.2 Disable the NAT Port Forward

Blocking the specific attacker IP is a short-term measure — the attacker could simply rotate to a different IP. The more durable response is to remove the exposed attack surface entirely by disabling the NAT rule that was forwarding RDP traffic from the WAN.

```
Firewall → NAT → Port Forward → disable TCP 3389 rule
```

<img width="1154" height="737" alt="image" src="https://github.com/user-attachments/assets/75f07c16-b899-40e5-9288-3ab9e362629b" />

<img width="1168" height="332" alt="image" src="https://github.com/user-attachments/assets/a23d656e-b9ed-4ab6-a23e-8d5c724e6190" />

This effectively closes the door regardless of where subsequent attempts originate. In a production environment, RDP access should be routed through a VPN rather than exposed via port forwarding — this incident is a direct consequence of that misconfiguration.

---

### 7.3 Assess scope of compromise

With the attacker's access cut off, the next question is whether the attack succeeded before the alert fired. A bruteforce alert tells you an attack was attempted — it does not tell you whether it worked. Checking for Event 4624 is the critical step that determines whether this is a containment exercise or a full incident response.

powershell

```powershell
# On DC01 — check for successful logons from the attacker IP
Get-WinEvent -FilterHashtable @{LogName='Security'; Id=4624} |
  Where-Object {$_.Message -like "*d.torres*"} |
  Select-Object TimeCreated, Message |
  Format-List
```

<img width="877" height="684" alt="image" src="https://github.com/user-attachments/assets/8d08ee51-b5a7-4eeb-8c72-e899bad5fe06" />

The steps above cover the immediate containment phase. A full incident response would go further — forensic analysis of WKS01, reviewing Sysmon telemetry for suspicious process activity post-logon, checking for persistence mechanisms (scheduled tasks, new local accounts, registry modifications), and correlating activity across all domain systems to rule out lateral movement. These are out of scope for this scenario but are directly addressed in the subsequent scenarios, where post-exploitation activity becomes the focus.

### Response checklist

- Source IP blocked in pfSense
- NAT Port Forward disabled
- Verified Event 4624 — confirmed presence or absence of successful logon
- Password reset for d.torres if successful logon was confirmed
- Reviewed WKS01 activity post-attack for signs of lateral movement (unusual outbound connections, new processes, Sysmon events)
- Documented timeline: first attempt → alert → response actions

## 8. Remediation

### 8.1 Root cause

The attack was made possible by a single misconfiguration, RDP exposed directly to the Internet via NAT Port Forward, with no authentication layer in front of it and no account lockout policy to limit credential attempts. Everything else, the bruteforce, the successful logon is a direct consequence of that.

---

### 8.2 What was implemented in this scenario

#### 8.2.1 Account Lockout Policy

Two controls were applied as part of the remediation demonstration, and their impact is visible directly in the logs.

**Account Lockout Policy** — enforcing a lockout threshold of 5 attempts stops the bruteforce before the attacker can exhaust the wordlist. The transition from SubStatus `0xC000006A` to `0xC0000234` in the 4625 events, followed by Event 4740, confirms the policy is working as intended. No 4624 follows — the attacker never gets in.

Account Lockout Policy was enabled via Default Domain Policy on DC01. Enforcing a lockout threshold of 5 attempts stops the bruteforce before the attacker can exhaust the wordlist.

In a real investigation, this helps establish whether the lockout occurred before or after a successful logon, which directly affects the severity assessment.

#### 8.2.3 Enabling NLA

**NLA (Network Level Authentication)** — re-enabling NLA removes the pre-authentication information exposure demonstrated during the reconnaissance phase. With NLA enforced, the login page is no longer reachable without valid credentials, the hostname and last logged-on username are no longer exposed, and unauthenticated screenshot capture via NetExec is no longer possible.

These two controls combined significantly raise the cost of the attack. The lockout policy caps the number of attempts an attacker can make before being blocked. NLA eliminates the reconnaissance advantage that made targeting straightforward in the first place.

### 8.3 Going further — a complete hardening posture

The controls above reduce risk but do not eliminate it. A determined attacker with a valid credential list and enough time can still succeed against a locked-out account once the lockout resets. The only complete solution is to remove the exposed attack surface entirely.

**Remove direct RDP exposure** — the NAT Port Forward should not exist in production. RDP should never be reachable from the Internet without a prior authentication layer. This is the single most impactful change.

**VPN-first access** — RDP becomes reachable only after authenticating through a VPN (WireGuard, OpenVPN). The RDP port is invisible to an external attacker who has not already authenticated to the VPN. Combined with MFA on the VPN, this reduces the RDP attack surface to near zero.

**Geo-blocking** — if remote access is legitimately used only from specific regions, pfSense GeoIP filtering can block authentication attempts from unexpected countries entirely, reducing noise and narrowing the attack window.

#### Detection improvements

Beyond the custom rules already implemented, a few additional detection opportunities are worth noting for future iteration.

Alerting on NTLM authentication events with no associated workstation name — as observed during testing with NetExec — would flag stealthy tooling that deliberately avoids exposing identifying information. A honeypot account (a credible-looking but unused account such as `admin` or `helpdesk`) would generate zero false positives by definition — any authentication attempt against it is suspicious and warrants immediate investigation. Finally, tuning alert thresholds by time of day — treating 3 failures at 2am as equivalent in severity to 8 failures during business hours — would improve signal quality without increasing false positive rates.
